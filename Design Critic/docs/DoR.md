# 🧩 DoR: ANTES de la implementación

Esta página reúne los criterios de calidad que deben cumplirse antes de considerar un diseño listo para desarrollo.

El objetivo es asegurar que los diseños entregados a los squads estén completos, claros y alineados al design system o productos de la tribu, reduciendo retrabajos, inconsistencias visuales y dudas durante la implementación.

---

## ✅ Antes de pasar a desarrollo (Checklist UX/UI)

Antes de mover un diseño a handoff o compartirlo con el squad, asegúrate de que el flujo cumpla con los siguientes criterios:

> (Tachar cuando ya esté completo)

---

## 🎯 Experiencia de usuario (UX)

- [ ] El flujo está completo y claro desde la perspectiva del usuario  
- [ ] No hay pantallas incompletas, duplicadas o faltantes  
- [ ] Se contemplan estados clave del flujo:
  - [ ] Estado vacío  
  - [ ] Error  
  - [ ] Loading  
  - [ ] Éxito / confirmación  
- [ ] Se consideraron escenarios o edge cases relevantes  
- [ ] El flujo tiene una lógica clara de navegación entre pantallas  

---

## 🎨 Diseño visual (UI)

El diseño debe ser consistente con el design system y las guías visuales del producto.

- [ ] Se prioriza el uso de componentes ya existentes  
- [ ] Tipografía, colores y estilos siguen las guías visuales definidas  
- [ ] Espaciados y alineaciones consistentes con la grid del producto  
- [ ] Iconos, botones y formularios mantienen consistencia visual  
- [ ] No se crearon componentes nuevos innecesarios sin validación previa  
- [ ] Estados de componentes (error, hover, disabled) están contemplados  
- [ ] No hay errores visuales evidentes  

---

## ✍️ Contenido y microcopy

Los textos deben ser claros, consistentes y útiles para el usuario.

- [ ] Los textos están completos (sin placeholders)  
- [ ] Los mensajes de error o ayuda están definidos  
- [ ] El tono del contenido es consistente con el producto  
- [ ] Etiquetas, botones y mensajes son claros y comprensibles  

---

## 🔗 Prototipo y especificaciones

El diseño debe ser comprensible para el squad.

- [ ] El prototipo está actualizado y navegable en Figma  
- [ ] Las interacciones principales están definidas  
- [ ] Se incluyen anotaciones cuando el comportamiento no es evidente  
- [ ] Los componentes están correctamente organizados en Figma  

---

## 🚀 Diseño listo para handoff

Cuando el flujo cumple con el checklist anterior:

1. El diseño puede considerarse **Design Ready**  
2. Puede compartirse mediante el proceso de handoff en Teams  
3. El squad puede iniciar la implementación con claridad  

---

## 💡 Buenas prácticas

- Revisar siempre el diseño antes de compartirlo con el squad  
- Priorizar el uso de componentes ya existentes para mantener consistencia  
- Evitar entregar flujos incompletos o con estados no definidos  
- Comunicar cambios importantes después del handoff  
- Socializar dudas visuales o de experiencia con el equipo de diseño
